# 生命演化骨架首期范围

本文件用于固定“生命演化骨架”首期工作的边界，避免在第一轮就滑向百科式铺量，或过早被外部数据源牵着走。

## 首期目标

首期目标不是补全所有纪元、类群和物种，而是先让资料库具备一套可稳定扩展的骨架：

- 全时期时间骨架：覆盖前寒武纪、古生代、中生代、新生代
- 主干类群骨架：支持后续从早期生命一路串到植物、动物、脊索动物、羊膜动物、鸟类、哺乳类和灵长类
- 最小关系模型：让时间页、支系页、事件页、代表卡和来源入口之间有统一互链
- 明确推进顺序：后续按阶段推进，而不是多条线同时发散

## 首期明确不做

为了保持骨架优先，本阶段明确不做以下内容：

- 不追求每个纪元都立刻补大量代表物种
- 不把 `04-species-cards/` 扩成百科式名录
- 不把 `06-registry/` 直接升级成数据库或自动抓取管线
- 不在第一阶段接入 `PBDB`
- 不以外部数据格式反推本地 schema

## PBDB 在首期的位置

首期先把 `PBDB` 视为“未来可接入来源”，而不是当前主导结构设计的核心来源。

这样安排的原因：

- 当前更紧迫的问题是统一本地 `id`、时间字段、关系字段和页面职责
- 若过早围绕外部接口设计，容易把第一轮时间花在字段映射与数据清洗，而不是把资料库骨架定稳
- 等本地 registry 稳定后，再判断 `PBDB` 更适合做校验源、补证源还是导入源

因此，首期只做两件与未来外部来源兼容的准备：

- 预留外部来源 ID 的承载位置
- 保持时间、分类、证据、争议字段的命名稳定

## 首期交付物

首期交付物只看“结构是否完整”，不看“条目数量是否很多”。

### 1. 时间骨架

- 宙、代、纪三层主框架明确
- `01-timeline/` 的时间页具备连续主线
- 代级总览与纪级页面之间的上下位关系写清

### 2. 支系主干

- `02-clades/` 中的大支系入口足够支撑主线
- 支系页不重复承担时间页职责
- 支系关系可被 `relation-index.md` 明确表达

### 3. 事件锚点

- 关键演化转折拥有稳定入口
- 事件页能同时挂回时间主轴和支系主线

### 4. Registry 最小集

- `entity-index.md` 负责条目身份
- `timeline-index.md` 负责时间窗口
- `relation-index.md` 负责主线关系
- `tag-index.md` 负责反向检索

## 首期推荐推进顺序

首期按以下顺序推进：

1. 先统一时间骨架和页面职责
2. 再补主干类群和事件锚点
3. 最后再用少量代表卡验证整套骨架可用

如果按纪元逐段推进，建议采用下面的批次，而不是平均铺开：

1. 前寒武纪 + 寒武纪
2. 泥盆纪 + 石炭纪 + 二叠纪
3. 三叠纪 + 侏罗纪 + 白垩纪
4. 古近纪 + 新近纪 + 第四纪

## 与现有目录的分工

- `00-index/`：定义规则、边界和骨架说明
- `01-timeline/`：承接全时期时间主线
- `02-clades/`：承接主干类群
- `03-events/`：承接转折事件
- `04-species-cards/`：承接少量关键代表卡
- `06-registry/`：承接结构化事实与反向索引
- `09-workbench/`：承接阶段推进矩阵、检查表和问题入口

## 首期完成判断

满足以下条件时，可认为首期骨架成立：

- 时间主轴已经覆盖从前寒武纪到第四纪的主干层级
- 各目录职责没有明显互相吞并
- 新增纪级页面知道应挂哪些时间、关系和来源字段
- 后续接入 `PBDB` 时，不需要推翻本地命名和索引结构
