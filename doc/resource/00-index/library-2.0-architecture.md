# 资料库 2.0 信息架构

本文件用于定义资料库 2.0 的目录职责边界。它的目标不是替代正文目录，而是把“正文、规则、登记、素材、导出、工作台”拆清，避免后续继续把所有辅助内容都堆进 `05-sources/`。

## 核心原则

- `01-04` 继续作为正文主库，不推翻现有按条目类型分层的组织方式。
- “纲 / 章 / 节 / 条目”的正式感，优先通过索引映射与 frontmatter 实现，而不是先强改物理目录。
- `05-sources/` 逐步收束为“来源入口层”，不再长期兼任规则层、工作台层和登记层。
- 结构化查询依赖 `06-registry/`，而不是依赖正文中的自然语言列表。
- 图片、图示、模型等素材统一挂接到 `07-assets/`。
- HTML timeline、游戏、3D 展示等消费需求统一预留到 `08-exports/`。

## 目录职责表

### `00-index/`

- 负责全局规则、命名规范、标签体系、地质年代骨架和 2.0 架构说明。
- 适合放“怎么组织资料库”的规则，不适合放具体专题矩阵或材料排查清单。

### `01-timeline/`

- 负责时间主轴正文。
- 页面重点回答“什么时候发生了什么，以及它处在怎样的长时段框架中”。

### `02-clades/`

- 负责演化支系正文。
- 页面重点回答“谁在演化树或组织性主线中承担关键位置”。

### `03-events/`

- 负责事件与转折正文。
- 页面重点回答“哪些机制或环境变化改变了后续生命史”。

### `04-species-cards/`

- 负责代表物种卡、结构型案例卡、过渡型案例卡。
- 页面重点回答“哪些可落地材料最适合被写成具体案例”。

### `05-sources/`

- 负责来源入口层。
- 适合保留权威网站入口、核心书目、来源总览。
- 过渡阶段仍可能暂存部分来源相关规范文件，但 2.0 目标是让它逐步收束为“去哪里找资料”的目录。

### `06-registry/`

- 负责结构化登记与索引。
- 适合放实体注册表、时间索引、标签索引、关系索引、来源登记。
- 它承担的是“可计算层”，不是正文层。

### `07-assets/`

- 负责素材与附件。
- 适合放图片、图示、模型、截图、时间轴素材、版权和来源说明。
- 正文不应直接承担素材库职责。

### `08-exports/`

- 负责导出与消费格式说明。
- 适合放 HTML timeline、游戏数据、3D 节点数据等对外消费结构。
- 它是派生层，不应反向污染正文和 registry。

### `09-workbench/`

- 负责研究与写作工作台工具。
- 适合放 checklist、matrix、问句入口、主题入口、专题工具图。
- 这类文件服务于“怎么查、怎么写、怎么排查”，不属于 source 本体。

### `templates/`

- 负责条目模板和 source-tool 模板。
- 它应同时服务正文层和部分工具层，但不承载实例内容。

## 纲 / 章 / 节 / 条目映射

### 纲

- 由 `01-timeline/`、`02-clades/`、`03-events/`、`04-species-cards/` 四类正文目录承担。

### 章

- 由目录内的主题组或索引映射承担。
- 例如前寒武纪主线、登陆主线、大灭绝主线等。

### 节

- 由单篇正文页面承担。
- 例如 `origin-of-life.md`、`great-oxidation-event.md`、`tiktaalik-roseae.md`。

### 条目

- 由页面内部的结构化段落承担。
- 例如“基本信息”“核心来源”“争议与说明”等模板段落。

## 迁移原则

### 第一阶段：定义优先

- 先明确新目录职责，不急着一次性移动所有旧文件。
- 先让根目录和 `00-index/` 能解释 2.0 的新结构。

### 第二阶段：最清晰的文件先迁

- `source-registry.md` 优先迁往 `06-registry/`。
- `checklist-*.md` 与 `matrix-*.md` 优先迁往 `09-workbench/`。

### 第三阶段：规则与模板再收束

- `citation-rules.md`、`controversy-recording-rules.md`、`evidence-layer-template.md` 逐步并入更清晰的规则层或在 `00-index/` 建立入口。

## 当前建议目录

```text
doc/resource/
  00-index/
  01-timeline/
  02-clades/
  03-events/
  04-species-cards/
  05-sources/
  06-registry/
  07-assets/
  08-exports/
  09-workbench/
  templates/
```

## 为什么先不直接上数据库

- 当前正文已经形成较强的 Markdown 写作惯性，直接迁库会把迁移成本放到最前面。
- 真正急需补的是身份字段、关系字段、时间字段和素材挂接点，而不是立即替换正文载体。
- 先把 Markdown 正规化，再让 registry 成为中间层，后续迁移 SQLite、JSON 或网页检索都会更稳。
